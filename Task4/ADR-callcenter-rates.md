### **Нефункциональные требования**

|**№**|**Требование**|
| :-: | :- |
| NFR-1 | **Актуальность ставок.** Ставки в системе кол-центра должны обновляться в течение минут после изменения в АБС (eventual consistency, допустимая задержка – до 5 минут). |
| NFR-2 | **Актуальность файлов на SFTP.** Файл со ставками обновляется автоматически при каждом изменении ставок в АБС. Партнёрский кол-центр забирает файлы по своему расписанию. |
| NFR-3 | **Доступность Deposit API Service 99,9%.** API ставок должен быть доступен для системы кол-центра круглосуточно (оператор может обслуживать клиента в любое время). |
| NFR-4 | **Формат файла для партнёра.** CSV или Excel – простой, понятный формат, совместимый с любой системой. Формат согласуется с партнёром. |
| NFR-5 | **Безопасность SFTP.** Доступ к SFTP-серверу ограничен IP-адресами партнёра. Аутентификация по ключам SSH. Ставки не являются персональными данными, но должны быть защищены от подмены. |
| NFR-6 | **Использование существующих технологий.** SFTP – стандартный протокол, не требует дополнительной экспертизы. Kafka Consumer для системы кол-центра – на Java (экспертиза есть). |
| NFR-7 | **Не увеличивать нагрузку на АБС.** Ставки распространяются через Kafka – АБС не получает дополнительных синхронных запросов от кол-центров. |

### **Решение**

Диаграммы C4 приведены в отдельных файлах:
- **Диаграмма контекста (C4 Level 1):** `c4-context-callcenter-rates.puml`
- **Диаграмма контейнеров (C4 Level 2):** `c4-containers-callcenter-rates.puml`

#### Архитектурное решение

Решение основано на архитектуре MVP из Task 3 и расширяет её двумя новыми каналами доставки ставок.

**Принцип: единый источник ставок → множественные потребители через Kafka.**

Модуль ставок в АБС – единственный источник правды (source of truth). Обновления ставок публикуются в Kafka (топик `deposit-rates`) через Kafka-адаптер АБС. Потребители подключаются к топику независимо:

```
АБС (модуль ставок)
    → Kafka-адаптер АБС
        → Kafka (deposit-rates)
            → Deposit API Service (кэш ставок)     → ИБ, Сайт [уже есть из Task 3]
            → Система кол-центра (Kafka Consumer)   → UI оператора [НОВОЕ]
            → Rates File Exporter                   → SFTP → Партнёрский КЦ [НОВОЕ]
```

**1. Доставка ставок в систему кол-центра банка**

Система кол-центра (React.js / Java Spring Boot / PostgreSQL, микросервисная) расширяется:
- **Новый Kafka Consumer** (Java / Spring Boot) – подключается к топику `deposit-rates`, получает обновления ставок и сохраняет их в БД системы кол-центра (PostgreSQL).
- **Новая страница в UI** (React.js) – раздел «Депозитные ставки» для операторов.
- **Альтернативно:** система кол-центра может запрашивать ставки у Deposit API Service по REST API (pull-модель). Kafka Consumer – предпочтительнее (push-модель), так как обеспечивает мгновенное обновление и не создаёт нагрузку на Deposit API.

Реализация силами Java-специалистов из команды АБС (есть Java-экспертиза + опыт доработки системы кол-центра).

**2. Доставка ставок в партнёрский кол-центр (SFTP)**

Партнёрский кол-центр – внешняя система, не имеющая API-интеграции с банком. Готов получать ставки в виде файлов через SFTP.

Решение:
- **Rates File Exporter** – новый микросервис (Java / Spring Boot), который:
  1. Подписывается на топик `deposit-rates` в Kafka.
  2. При получении обновления генерирует файл со ставками (CSV).
  3. Выкладывает файл на SFTP-сервер банка.
- **SFTP-сервер** – разворачивается в инфраструктуре банка (IT-отдел). Стандартный Linux + OpenSSH.
- Партнёрский кол-центр забирает файлы с SFTP по своему расписанию.

**Почему отдельный микросервис (Rates File Exporter), а не часть Deposit API Service:**
- Разделение ответственности: Deposit API Service обслуживает онлайн-каналы (сайт, ИБ), а экспорт файлов – другая задача с другим SLA.
- Независимое масштабирование и деплой.
- При сбое SFTP-экспорта не затрагивается онлайн-обслуживание клиентов.

**3. Технологический стек новых компонентов**

| Компонент | Технология | Обоснование |
|-----------|-----------|-------------|
| Kafka Consumer (КЦ) | Java / Spring Boot | Экспертиза в команде, совместимость с системой КЦ |
| Rates File Exporter | Java / Spring Boot | Единый стек, экспертиза в команде |
| SFTP-сервер | Linux / OpenSSH | Стандартное решение, поддержка IT-отделом |
| Формат файла | CSV | Простой, универсальный, согласован с партнёром |

### **Альтернативы**

**Альтернатива 1: REST API Deposit API Service → Система кол-центра (pull-модель)**

Система кол-центра запрашивает ставки у Deposit API Service по REST API при каждом обращении оператора, вместо Kafka Consumer.

Преимущества:
- Проще реализация – не нужен Kafka Consumer в системе кол-центра.
- Всегда актуальные данные (запрос в реальном времени).

Почему отклонено:
- **Связанность:** система кол-центра зависит от доступности Deposit API Service. При сбое API – операторы не видят ставки.
- **Нагрузка:** при 200+ операторах каждое обращение к клиенту генерирует запрос к API. Kafka Consumer кэширует данные локально.
- **Задержка:** REST-запрос добавляет latency к UI оператора.

**Альтернатива 2: Прямая интеграция системы кол-центра с АБС (API / БД)**

Система кол-центра обращается к АБС напрямую за ставками.

Преимущества:
- Источник данных – напрямую из АБС, без промежуточных звеньев.

Почему отклонено:
- **Нагрузка на АБС (NFR-7):** противоречит принципу изоляции АБС (NFR-3 из Task 3). БД Oracle перегружена.
- **Архитектурная несогласованность:** остальные системы уже интегрируются через Kafka. Прямая интеграция создаёт параллельный канал.

**Альтернатива 3: Email вместо SFTP для партнёрского кол-центра**

Отправка файлов со ставками по email партнёру.

Преимущества:
- Не нужен SFTP-сервер.
- Привычный канал коммуникации.

Почему отклонено:
- **Ненадёжность:** email может попасть в спам, задержаться.
- **Неавтоматизируемость:** на стороне партнёра сложнее автоматизировать обработку из email, чем с SFTP.
- **Безопасность:** email менее безопасен, чем SFTP с SSH-ключами.
- **Масштабируемость:** при увеличении числа партнёров SFTP проще масштабировать (разные директории).

### **Недостатки, ограничения, риски**

**Недостатки:**

1. **Дополнительная инфраструктура.** Добавляются SFTP-сервер и Rates File Exporter. Увеличивается количество компонентов для мониторинга.

2. **Задержка обновления для партнёрского кол-центра.** Партнёр забирает файлы по своему расписанию – возможно расхождение ставок на период между синхронизациями. Это приемлемо, так как ставки обновляются ежедневно.

3. **Формат файла требует согласования.** Необходимо согласовать с партнёром структуру CSV-файла, кодировку, формат дат и чисел.

**Ограничения:**

1. **Партнёрский кол-центр не имеет API.** Файловый обмен через SFTP – единственный доступный вариант интеграции.

2. **SFTP-сервер – зона ответственности IT-отдела.** Необходимо согласовать обслуживание и мониторинг.

**Риски:**

1. **Рассинхронизация ставок.** При сбое Kafka или потребителя ставки в кол-центре могут отстать от АБС. **Митигация:** health-check потребителей, алерты при отсутствии обновлений дольше 1 часа, fallback – запрос к Deposit API по REST.

2. **Партнёр не забирает файлы вовремя.** Операторы партнёрского кол-центра консультируют по устаревшим ставкам. **Митигация:** TTL-метка в файле (дата/время генерации), договорённость о частоте синхронизации (минимум раз в день).

3. **Безопасность SFTP-сервера.** Доступ извне – потенциальный вектор атаки. **Митигация:** IP-фильтрация, SSH-ключи, аудит доступа, размещение в DMZ.
