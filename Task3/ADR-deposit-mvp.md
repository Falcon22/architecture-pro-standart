### **Решение**

Диаграммы C4 приведены в отдельных файлах:
- **Диаграмма контекста (C4 Level 1):** `c4-context-deposit-mvp.puml`
- **Диаграмма контейнеров (C4 Level 2):** `c4-containers-deposit-mvp.puml`

#### Ключевые архитектурные решения

**1. Deposit API Service – новый микросервис (middleware)**

Центральное архитектурное решение – введение нового сервиса **Deposit API Service**, который выступает промежуточным слоем между онлайн-каналами (сайт, интернет-банк) и внутренними системами банка (АБС).

Обоснование:
- **Изоляция АБС (NFR-3, NFR-6):** БД АБС перегружена и масштабируется только вертикально. Прямая интеграция онлайн-каналов с АБС создаёт риск для работоспособности банка. Deposit API Service принимает все запросы от каналов и взаимодействует с АБС асинхронно через Kafka.
- **Независимость от подрядчика ИБ (NFR-8):** Бизнес-логика работы с депозитами (расчёт ставок, подача заявок, СМС-подтверждение) реализована вне монолита интернет-банка. Интернет-банк обращается к Deposit API Service по REST API.
- **Кэширование ставок (NFR-5):** Deposit API Service кэширует актуальные ставки из АБС, обеспечивая отклик в миллисекунды без нагрузки на АБС.
- **Горизонтальное масштабирование (NFR-6):** Сервис stateless и легко масштабируется горизонтально.

**Технологический стек Deposit API Service:**
- **Язык/фреймворк:** Java / Spring Boot – в банке есть экспертиза (команда АБС, система кол-центра), совместимость с Kafka (NFR-9).
- **БД:** PostgreSQL – для хранения заявок, кэша ставок и статусов. Экспертиза есть (система кол-центра на PostgreSQL).
- **Брокер:** Apache Kafka – для асинхронной интеграции с АБС и системой кол-центра (NFR-9, +R3).

**2. Apache Kafka как интеграционная шина**

Kafka используется для асинхронного взаимодействия между компонентами:
- **Deposit API Service → АБС:** заявки на депозиты (topic: `deposit-applications`).
- **АБС → Deposit API Service:** обновления ставок (topic: `deposit-rates`), статусы заявок (topic: `deposit-statuses`).
- **Deposit API Service → Система кол-центра:** заявки с сайта (topic: `website-applications`).

Обоснование:
- Гарантированная доставка (NFR-4): Kafka обеспечивает at-least-once delivery с персистентным хранением сообщений.
- Развязка систем: АБС, кол-центр и онлайн-каналы работают в своём темпе.
- Перспективность (+R3): бизнес выбрал Kafka для будущих интеграций.

**АБС-сторона:** для чтения/записи в Kafka на стороне АБС реализуется интеграционный адаптер (Java-приложение), который транслирует сообщения из Kafka в хранимые процедуры Oracle PL/SQL и обратно. Это позволяет не модифицировать архитектуру АБС, а добавить «мост» к ней.

**3. Модуль управления ставками в АБС**

Текущий расчёт ставок в Excel заменяется модулем в АБС:
- Реализуется как набор PL/SQL-процедур и таблиц в Oracle (собственная разработка, команда АБС – 20 чел.).
- Поддерживает ролевой доступ: сотрудники бэк-офиса депозитов и кредитов работают в своих разделах (NFR-11).
- Обновления ставок публикуются в Kafka через интеграционный адаптер АБС.

**4. Модификация интернет-банка – минимальная**

В монолит интернет-банка (ASP.NET MVC 4.5) добавляются только:
- Новые страницы/контроллеры для UI депозитов (список, форма заявки).
- REST-вызовы к Deposit API Service для получения ставок и отправки заявок.
- Никакой бизнес-логики депозитов в монолите – всё делегируется Deposit API Service.

Это минимизирует зависимость от подрядчика (NFR-8) и не затрагивает ядро системы.

**5. СМС-подтверждение и уведомления**

- **Подтверждение заявки (СМС-код):** Deposit API Service генерирует и верифицирует OTP, отправляет через СМС-шлюз. Функционал реализован в Deposit API Service, а не в ядре ИБ (NFR-8).
- **Уведомления (подтверждение ставки, открытие депозита):** Deposit API Service слушает топик `deposit-statuses` из Kafka и отправляет СМС через шлюз.

**6. Сайт – минимальные доработки**

- Новая страница с формой заявки на депозит (PHP/React.js).
- REST-вызовы к Deposit API Service для получения ставок и отправки заявки.
- HTTPS обязателен (NFR-7).

**7. Развёртывание и отказоустойчивость**

- Deposit API Service и Kafka разворачиваются в основном ЦОД с репликацией в резервный.
- Балансировщик нагрузки распределяет запросы между экземплярами Deposit API Service (NFR-2, NFR-6).
- Kafka cluster – минимум 3 брокера для отказоустойчивости.

### **Альтернативы**

**Альтернатива 1: Прямая интеграция ИБ с АБС через API (REST/SOAP)**

Вместо Kafka – синхронная интеграция интернет-банка с АБС через REST API поверх Oracle.

Преимущества:
- Проще в реализации на начальном этапе.
- Нет необходимости разворачивать и поддерживать Kafka-кластер.
- Мгновенная консистентность данных.

Почему отклонено:
- **БД АБС перегружена** – синхронные запросы от онлайн-каналов усилят нагрузку и создадут риск для работоспособности банка (нарушение NFR-3).
- **АБС масштабируется только вертикально** – при росте числа онлайн-клиентов система не выдержит (NFR-6).
- **Не обеспечивает гарантированной доставки** – при сбое АБС заявки теряются (нарушение NFR-4).
- **Связанность** – интернет-банк становится жёстко зависим от доступности АБС.

**Альтернатива 2: Расширение монолита интернет-банка (бизнес-логика депозитов внутри ИБ)**

Реализовать логику работы с депозитами непосредственно в ASP.NET MVC приложении интернет-банка.

Преимущества:
- Один деплой, нет дополнительных сервисов.
- Меньше сетевых вызовов.

Почему отклонено:
- **ASP.NET MVC 4.5 несовместим с Kafka** (+R2) – невозможно реализовать асинхронную интеграцию.
- **Доработка ядра требует подрядчика** (NFR-8) – длительный цикл согласования и разработки.
- **Монолит не масштабируется горизонтально** для отдельных модулей (NFR-6).
- **Нет микросервисной архитектуры** – противоречит стратегическому направлению (S3).

**Альтернатива 3: RabbitMQ вместо Kafka**

Использовать RabbitMQ как брокер сообщений.

Преимущества:
- Проще в настройке для паттерна request-reply.
- Меньший порог входа для команды.

Почему отклонено:
- **Бизнес выбрал Kafka** (+R3) – решение зафиксировано как стратегическое для перспективных интеграций.
- Kafka лучше подходит для event-driven архитектуры и масштабируется горизонтально.
- Kafka обеспечивает replay сообщений (полезно для восстановления после сбоев).

### **Недостатки, ограничения, риски**

**Недостатки:**

1. **Увеличение количества инфраструктурных компонентов.** Добавляются Deposit API Service, Kafka-кластер, интеграционный адаптер АБС. Это увеличивает сложность развёртывания и мониторинга.

2. **Eventual consistency.** Данные между онлайн-каналами и АБС синхронизируются асинхронно – возможна задержка (секунды-минуты). Клиент может не сразу увидеть обновлённый статус заявки.

3. **Необходимость развёртывания Kafka.** В банке нет опыта эксплуатации Kafka. Потребуется обучение команды или привлечение специалистов. Kafka-кластер требует ресурсов и мониторинга.

**Ограничения:**

1. **MVP сохраняет ручную обработку бэк-офисом.** Менеджер бэк-офиса по-прежнему вручную обрабатывает заявки в АБС. Полная автоматизация – следующий этап (через 12 месяцев).

2. **Модуль ставок в АБС ограничен возможностями Oracle PL/SQL.** Сложные алгоритмы расчёта ставок могут потребовать дополнительных ресурсов Oracle и вертикального масштабирования.

3. **Новые клиенты с сайта всё равно должны прийти в отделение** для идентификации (регуляторное требование). Полностью онлайн-процесс для новых клиентов невозможен.

4. **Интернет-банк остаётся монолитом.** Новые UI-страницы добавляются в монолит ASP.NET MVC. Микросервисная архитектура применяется только для backend (Deposit API Service).

**Риски:**

1. **Риск перегрузки АБС при массовой публикации событий.** Интеграционный адаптер АБС читает из Kafka и вызывает PL/SQL процедуры. При большом потоке заявок может возникнуть нагрузка на Oracle. **Митигация:** rate limiting в адаптере, контроль пропускной способности, мониторинг нагрузки на Oracle.

2. **Отсутствие экспертизы по Kafka.** Внедрение новой технологии – риск для стабильности. **Митигация:** обучение команды, привлечение внешнего консультанта на этапе внедрения, постепенное наращивание нагрузки.

3. **Зависимость от СМС-шлюза.** При сбое шлюза клиент не получит OTP-код и не сможет подтвердить заявку. **Митигация:** мониторинг доступности шлюза, retry-механизм, информирование клиента о задержке.

4. **Несогласованность данных при длительном сбое Kafka.** Если Kafka-кластер недоступен, заявки будут накапливаться на стороне Deposit API Service. **Митигация:** локальное персистентное хранение неотправленных сообщений, алерты, Kafka-кластер из 3+ брокеров в двух ЦОД.

5. **Сроки реализации модуля ставок в АБС.** Команда АБС (20 чел.) загружена текущими задачами. Разработка модуля ставок может затянуться. **Митигация:** приоритизация задачи руководством, выделение отдельной подкоманды.
