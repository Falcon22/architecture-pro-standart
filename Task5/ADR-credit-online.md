#### Архитектурное решение

Решение строится на архитектуре, заложенной в MVP для депозитов (Task 3), и расширяет её для кредитного процесса. Используется та же инфраструктура (Kafka, два ЦОД) и тот же архитектурный подход (микросервис-посредник между онлайн-каналами и бэкендом).

**1. Credit API Service – новый микросервис**

Аналогично Deposit API Service, создаётся **Credit API Service** – микросервис, который:
- Принимает кредитные заявки от сайта и интернет-банка через REST API.
- Отдаёт список кредитных предложений (кэш из Кредитного конвейера) и предодобренные предложения.
- Публикует заявки в Kafka (топик `credit-applications`).
- Получает обновления статусов из Kafka (топик `credit-statuses`) и отправляет СМС-уведомления.
- Хранит результаты предрасчитанных скорингов для предодобренных предложений.
- Для мгновенного скоринга на сайте – вызывает Систему скоринга синхронно через REST API.

**Технологический стек:** Java / Spring Boot, PostgreSQL, Kafka. Аналогичен Deposit API Service – единый стек, единая экспертиза.

**Обоснование выделения в отдельный сервис (а не расширение Deposit API):**
- Разные домены (депозиты vs. кредиты) – разная бизнес-логика, разные SLA.
- Независимый деплой и масштабирование.
- Разные зависимости: Credit API зависит от Системы скоринга и Кредитного конвейера, Deposit API – от модуля ставок АБС.

**2. Решение проблемы пакетного обмена (NFR-4)**

Ключевая архитектурная проблема: заявки из онлайн-каналов должны попадать в Кредитный конвейер в тот же день, но пакетный обмен АБС→Конвейер через БД (1 раз/сутки) ускорить нельзя.

**Решение: прямой канал Credit API Service → Kafka → Кредитный конвейер.**

```
БЫЛО (offline):
  Отделение → АБС (Oracle) → [пакет, 1 раз/сутки] → Кредитный конвейер

СТАЛО (online заявки):
  ИБ / Сайт → Credit API Service → Kafka (credit-applications) → Кредитный конвейер
                                                                 ↓
                                                           АБС (синхронизация через Kafka
                                                                для фронт-офиса, UC-6)
```

- Онлайн-заявки идут напрямую в Кредитный конвейер через Kafka, **минуя АБС**.
- Кредитный конвейер получает заявку мгновенно (не ждёт суточного пакета).
- АБС получает копию заявки через Kafka (для отображения в десктоп-клиенте фронт-офиса), но это не блокирует обработку.
- Существующий пакетный обмен АБС→Конвейер сохраняется для заявок из отделений (legacy-процесс).

**Кредитный конвейер** (Camunda BPM / Java / Oracle) уже поддерживает горизонтальное масштабирование и работу 24/7 – готов к приёму заявок из Kafka. Необходимо разработать Kafka Consumer для Кредитного конвейера.

**3. Предодобренные предложения (UC-3, UC-7)**

Предодобренные предложения требуют предварительного скоринга, но система скоринга перегружена в рабочие часы (NFR-5).

**Решение: Pre-scoring Batch Job (ночной batch) + кэш в Credit API Service.**

- **Pre-scoring Batch Job** – отдельный Java-процесс, запускается по расписанию (cron) в нерабочие часы.
- Получает список активных клиентов ИБ из АБС (через Kafka-адаптер или напрямую из БД через read-only реплику).
- Для каждого клиента вызывает Систему скоринга (REST API) – скоринг с полными данными (АБС + Бюро КИ).
- Результаты (одобренная сумма, условия, срок действия) сохраняются в БД Credit API Service.
- При входе клиента в ИБ – Credit API отдаёт кэшированное предложение.

**Почему batch, а не realtime:**
- Система скоринга перегружена в рабочие часы (NFR-5) – realtime-предрасчёт добавит нагрузку.
- Предодобренное предложение не требует мгновенной актуальности – данные актуальны в пределах суток.
- Batch в ночное время утилизирует простаивающие мощности скоринга.

**4. Мгновенный скоринг на сайте (UC-2)**

Для новых клиентов с паспортными данными – синхронный вызов Системы скоринга:

```
Сайт → Credit API Service → Система скоринга (REST API)
                                    ↓
                              Бюро КИ (REST API, онлайн)
```

- Для нового клиента используются **только данные Бюро КИ** (внутренних данных нет).
- Вызов синхронный, но это допустимо: скоринг по одному клиенту (не массовый предрасчёт), данные только из Бюро КИ (быстрый ответ).
- SLA: ≤ 5 секунд (NFR-9).

**5. Сквозная заявка между каналами (UC-6, NFR-11)**

Заявка, созданная на сайте или в ИБ, должна быть доступна менеджеру в отделении (АБС).

**Решение:**
- Credit API Service присваивает каждой заявке уникальный `application_id`.
- Заявка публикуется в Kafka → Кредитный конвейер (для обработки) и АБС (для отображения фронт-офису).
- Kafka-адаптер АБС записывает заявку в Oracle с тем же `application_id`.
- Менеджер фронт-офиса находит заявку в АБС по телефону/ФИО/ID клиента и продолжает работу.

**6. СМС-информирование (UC-8)**

- Кредитный конвейер при изменении статуса заявки публикует событие в Kafka (топик `credit-statuses`).
- Credit API Service (Kafka Consumer) получает событие и отправляет СМС через SMS/OTP Service → СМС-шлюз.
- Необходим новый Kafka Producer в Кредитном конвейере – разработка Kafka-адаптера для Конвейера.

### **Альтернативы**

**Альтернатива 1: Расширение Deposit API Service для кредитов (единый сервис)**

Добавить кредитную функциональность в существующий Deposit API Service вместо создания отдельного Credit API Service.

Преимущества:
- Меньше сервисов для деплоя и мониторинга.
- Переиспользование SMS/OTP-логики, Kafka-инфраструктуры.
- Единая точка входа для онлайн-каналов.

Почему отклонено:
- **Нарушение Single Responsibility.** Депозиты и кредиты – разные домены с разными зависимостями (скоринг, конвейер vs. модуль ставок).
- **Blast radius.** Сбой в кредитном модуле повлияет на депозитный процесс и наоборот.
- **Независимое масштабирование.** Кредитный процесс (скоринг, предрасчёт) имеет другой профиль нагрузки.
- **Разные команды.** Депозитная и кредитная функциональность могут развиваться разными командами.

При этом SMS/OTP Service – общий для обоих сервисов (shared service).

**Альтернатива 2: Заявки через АБС (существующий пакетный процесс)**

Онлайн-заявки на кредит записываются в АБС (через Kafka-адаптер), а далее идут в Кредитный конвейер через существующий суточный пакет.

Преимущества:
- Не нужен Kafka Consumer в Кредитном конвейере.
- Единый поток заявок (все идут через АБС).

Почему отклонено:
- **Невозможно обработать заявку в тот же день (NFR-4).** Пакетный обмен – 1 раз/сутки, ускорить нельзя.
- **Предодобренные заявки теряют смысл** – клиент подаёт заявку утром, она попадает в конвейер только на следующий день.
- **Нарушение бизнес-требования** о решении по кредиту в течение одного дня.

**Альтернатива 3: Realtime-скоринг для предодобренных предложений (вместо batch)**

Рассчитывать предодобренное предложение в момент входа клиента в ИБ (realtime-вызов скоринга).

Преимущества:
- Всегда актуальные данные.
- Не нужен Pre-scoring Batch Job.

Почему отклонено:
- **Система скоринга перегружена в рабочие часы (NFR-5).** Массовый realtime-скоринг при каждом входе клиента создаст неприемлемую нагрузку.
- **Время отклика.** Скоринг с запросом в Бюро КИ может занимать секунды – задержка при открытии страницы ИБ.
- **Нецелесообразно.** Скоринг не меняется каждую минуту – ночной batch достаточен для актуальности в пределах суток.

**Альтернатива 4: Объединение Credit API Service и Кредитного конвейера**

Подавать заявки напрямую в Кредитный конвейер (Camunda) из онлайн-каналов, без промежуточного Credit API.

Преимущества:
- Меньше компонентов.
- Заявка сразу в системе обработки.

Почему отклонено:
- **Кредитный конвейер – коробочное решение** с ограниченными возможностями расширения. Добавление REST API для внешних клиентов может быть сложным.
- **Связанность.** Онлайн-каналы напрямую зависят от доступности Конвейера.
- **Нет места для предодобренных предложений.** Конвейер обрабатывает заявки, а не управляет кэшем предложений.
- **Безопасность.** Конвейер содержит внутренние данные – прямой REST API наружу нежелателен.

### **Недостатки, ограничения, риски**

**Недостатки:**

1. **Увеличение количества сервисов.** Credit API Service, Pre-scoring Batch Job, Kafka Consumer для Конвейера – три новых компонента. Увеличивается сложность мониторинга и эксплуатации.

2. **Eventual consistency заявок.** Заявка, созданная на сайте, появится в АБС (для фронт-офиса) с задержкой в секунды-минуты. Менеджер может не увидеть заявку в первые минуты после её создания.

3. **Предодобренные предложения актуальны в пределах суток.** Batch-скоринг запускается ночью – в течение дня ситуация клиента может измениться. Поэтому при подаче заявки по предодобренному предложению выполняется повторный упрощённый скоринг.

4. **Два канала доставки заявок в Конвейер.** Онлайн-заявки идут через Kafka, заявки из отделений – через суточный пакет из АБС. Два разных пути создают сложность в администрировании и мониторинге.

**Ограничения:**

1. **Новые клиенты с сайта – идентификация только в отделении.** Даже при одобренном скоринге клиент должен прийти в отделение (регуляторное требование).

2. **Мгновенный скоринг на сайте – только по данным Бюро КИ.** Для нового клиента нет внутренних данных банка. Точность скоринга ограничена.

3. **Пакетный обмен АБС→Конвейер сохраняется для legacy-процесса.** Заявки из отделений продолжают идти суточным пакетом. Унификация потоков – задача будущих итераций.

4. **Pre-scoring Batch Job – временное окно.** Количество клиентов для предрасчёта ограничено мощностью скоринговой системы в ночные часы. При росте клиентской базы может потребоваться приоритизация (скоринг только для активных клиентов).

5. **Кредитный конвейер – коробочное решение.** Добавление Kafka Consumer возможно, но требует координации с подрядчиком, внедрявшим систему. В IT-отделе есть специалисты, способные менять обработчики и описания бизнес-процессов.

**Риски:**

1. **Нагрузка на скоринг от мгновенных запросов с сайта.** При успешной маркетинговой кампании количество заявок с паспортными данными может быть значительным. **Митигация:** rate limiting на Credit API Service, мониторинг нагрузки скоринга, circuit breaker (при перегрузке – возвращать «заявка принята, решение позже» вместо мгновенного скоринга).

2. **Конфликт данных при сквозной заявке.** Менеджер в отделении может начать редактировать заявку, которая одновременно обрабатывается в Конвейере. **Митигация:** статусная модель заявки – если заявка уже в обработке, менеджер видит статус «на рассмотрении» и не может её изменить.

3. **Сбой Pre-scoring Batch Job.** Если ночной batch не отработал, клиенты не увидят предодобренных предложений. **Митигация:** мониторинг и алерты, retry-логика, fallback – показывать общие предложения вместо предодобренных.

4. **Безопасность паспортных данных.** Передача и хранение ПДн создаёт регуляторные риски. **Митигация:** шифрование at-rest и in-transit, минимизация хранения (удаление после обработки), аудит доступа, compliance review.

5. **Координация с подрядчиком Кредитного конвейера.** Разработка Kafka Consumer для Конвейера может потребовать участия подрядчика. **Митигация:** использовать внутреннюю экспертизу (IT-отдел умеет менять обработчики Camunda), привлечь подрядчика только для ревью.
