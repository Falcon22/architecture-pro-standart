| F2 | **Сайт: подача заявки на депозит (Ф. И. О. + телефон).** После подачи заявки менеджер кол-центра перезванивает клиенту. | Заявка должна попадать в систему кол-центра. Необходима новая интеграция сайт → система кол-центра. После звонка клиент приходит в отделение для идентификации. |

О. + телефон + опционально паспортные данные).** При заполнении паспортных данных – мгновенное предварительное решение (одобрение/отказ). | Для предварительного решения нужен вызов системы скоринга, которая обращается в Бюро КИ по REST API. Необходима автоматизация расчёта ставок, который сейчас выполняется вручную в Excel. Архитектурно значимо: нужен сервис расчёта ставок. |
| F7 | **Интернет-банк: подача заявки на депозит (выбор счёта, суммы) с подтверждением по СМС-коду.** | Функционал СМС реализован в ядре интернет-банка (подрядчик). Бизнес-логику подтверждения нужно реализовать силами команды банка, без доработки ядра. Архитектурно значимо: ограничение на модификацию ядра интернет-банка. |
| F8 | **Интернет-банк: обработка заявки на депозит менеджером бэк-офиса (MVP).** Менеджер подтверждает условия депозита в АБС. Клиент получает СМС после подтверждения ставки и после открытия депозита. | Этап MVP: процесс остаётся частично ручным. Заявка из ИБ должна попадать в АБС для обработки бэк-офисом. Необходимо избегать прямой интеграции ИБ → АБС (см. R3). |
| F9 | **Интернет-банк: отображение кредитных предложений и предодобренных предложений.** Предодобренное предложение означает, что скоринг уже произведён и рассчитана приемлемая сумма кредита. | Требует хранения результатов предварительного скоринга и механизма их доставки в ИБ. Архитектурно значимо: нужен источник данных о предодобренных предложениях, отдельный от нагруженной системы скоринга. |
| F10 | **Интернет-банк: подача заявки на кредит (счёт зачисления, данные заявки) с подтверждением по СМС-коду.** При подаче по предодобренному предложению – повторный скоринг по упрощённой процедуре. | Аналогично F7: СМС-подтверждение без доработки ядра подрядчика. Повторный скоринг должен использовать упрощённую процедуру, чтобы не перегружать систему скоринга. |
| F11 | **Обеспечение непрерывности заявки на кредит между каналами.** Если клиент получил предварительное одобрение на сайте и пришёл в отделение, менеджер должен продолжить работу с той же заявкой. | Архитектурно значимо: заявки должны храниться в едином месте, доступном и сайту, и фронт-офису (АБС), и Кредитному конвейеру. Требуется сквозной идентификатор заявки. |
| F12 | **Модуль управления ставками депозитов в АБС.** Замена Excel-файлов на функциональность в АБС. Доступ для сотрудников бэк-офиса депозитов и бэк-офиса кредитов. | Устраняет ручной процесс передачи ставок по email. Является предпосылкой для автоматизации расчёта ставок в онлайн-каналах. Архитектурно значимо: расширение АБС. |
| F13 | **СМС-информирование клиента об изменении статуса кредитной заявки.** | Требует механизма отслеживания статусов заявок и триггеров для отправки СМС через шлюз. Кредитный конвейер должен уметь инициировать отправку СМС. |
| F14 | **Обработка предодобренных кредитных заявок в течение одного рабочего дня.** Текущий пакетный обмен АБС → Кредитный конвейер (1 раз/сутки) не позволяет это обеспечить. | Архитектурно значимо: необходим альтернативный канал доставки заявок в Кредитный конвейер, минуя пакетный обмен через БД АБС. Ключевое архитектурное решение. |
| F15 | **Автоматизация процесса согласования ставки по депозитной заявке.** Перенос текущего процесса (email между бэк-офисом депозитов, отделом кредитования и фронт-офисом) в АБС. | Подготовка к полной автоматизации (цель на 12 мес.). На этапе MVP может остаться частично ручным, но модуль ставок (F12) должен быть реализован. |
| **U** | **Удобство использования (Usability)** | |
| U1 | **Единая система дизайна.** Интерфейсы интернет-банка и сайта должны следовать корпоративной дизайн-системе: брендированные элементы, фирменные цвета, единообразные UI-компоненты. | Явное требование бизнеса. Обеспечивает узнаваемость и доверие клиентов. |
| U2 | **Максимально удобный интерфейс для клиента.** Интуитивная навигация, минимум шагов для подачи заявки, понятные формы. | Бизнес-ожидание. Для привлечения молодой аудитории UX должен соответствовать уровню конкурентов. |
| U3 | **Быстрый отклик интерфейса – операции в миллисекундах.** Справочные данные не должны загружаться дольше секунды (текущая проблема). | Архитектурно значимо: требует кэширования справочных данных на стороне интернет-банка, а не прямых запросов к перегруженной БД АБС. Влияет на выбор архитектуры интеграции. |
| **R** | **Надёжность (Reliability)** | |
| R1 | **Доступность сервисов 24/7 с SLA 99,9%.** Все онлайн-сервисы (интернет-банк, сайт) должны быть доступны круглосуточно. | 99,9% = не более 8,76 часов простоя в год. Архитектурно значимо: требует отказоустойчивой архитектуры, резервирования, health checks. |
| R2 | **Отказоустойчивость при сбое ЦОД.** При сбое основного ЦОД интернет-банк должен продолжать работу на резервном ЦОД с сохранением требуемой нагрузки. | У банка есть резервный ЦОД, но сейчас всё работает на одной группе серверов. Архитектурно значимо: требует active-standby или active-active конфигурации между ЦОД, механизмов переключения. |
| R3 | **Изоляция интернет-банка от прямой нагрузки на АБС.** Онлайн-подача заявок не должна создавать риск для работоспособности АБС. Избегать прямой работы ИБ с API/БД АБС в новых процессах. | БД АБС уже перегружена (подтверждено командой АБС). Архитектурно значимо: необходим промежуточный слой (middleware/очередь) между онлайн-каналами и АБС для асинхронного обмена данными. Ключевое архитектурное решение. |
| R4 | **Устойчивость к потере заявок.** Заявки на депозиты и кредиты, поданные онлайн, не должны теряться при сбоях промежуточных систем. | Дополнительное требование. Финансовые заявки критичны – потеря заявки = потеря клиента и репутационный ущерб. Требует гарантированной доставки сообщений (at-least-once). |
| R5 | **Консистентность данных между каналами.** Клиент и сотрудник должны видеть одинаковый статус заявки вне зависимости от канала (сайт, ИБ, отделение, кол-центр). | Дополнительное требование, следует из F11. Eventual consistency допустима с задержкой не более нескольких минут. |
| **P** | **Производительность (Performance)** | |
| P1 | **Время отклика операций интернет-банка – миллисекунды.** Справочные данные должны загружаться менее чем за 1 секунду (исправление текущей проблемы). | Требует введения кэширования на стороне ИБ. Архитектурно значимо: влияет на выбор паттерна интеграции (синхронный vs. асинхронный с кэшем). |
| P2 | **Горизонтальное масштабирование интернет-банка и равномерное распределение нагрузки** между серверами, приложениями и ЦОД. | Архитектурно значимо: текущий монолит ASP.NET MVC 4.5 сложно масштабировать горизонтально. Переход к микросервисной архитектуре (хотя бы для новых модулей) и использование балансировщика нагрузки. |
| P3 | **АБС масштабируется только вертикально** из-за архитектуры базы данных (Oracle PL/SQL с процедурной логикой). | Жёсткое ограничение. Подтверждает необходимость промежуточного слоя (R3) для защиты АБС от роста онлайн-нагрузки. |
| P4 | **Система скоринга не должна нагружаться дополнительными процессами предрасчёта скорингов.** Система испытывает повышенную нагрузку в рабочие часы. | Архитектурно значимо: предодобренные предложения должны рассчитываться асинхронно в нерабочие часы (batch) и кэшироваться. Запрещено выполнять массовый предрасчёт в рабочее время. |
| P5 | **Кредитный конвейер и система скоринга поддерживают горизонтальное масштабирование и работу 24/7.** | Позитивное ограничение: эти системы готовы к росту нагрузки. Новые интеграции могут опираться на их масштабируемость. |
| P6 | **Предодобренные кредитные заявки обрабатываются в течение одного рабочего дня.** При этом ускорить пакетный обмен АБС → Кредитный конвейер невозможно. | Архитектурно значимо: для предодобренных заявок нужен прямой канал в Кредитный конвейер, минуя АБС. Это влияет на архитектуру нового сервиса подачи заявок. |
| **S** | **Поддерживаемость (Supportability)** | |
| S1 | **Использование существующих технологий банка при доработках:** БД – MS SQL, Oracle; языки и платформы, знакомые команде. | Снижает риски внедрения и стоимость сопровождения. Новые технологии допустимы при совместимости с существующими платформами и наличии внутренней экспертизы. |
| S2 | **Экспертиза в используемых технологиях.** Внутри банка должны быть специалисты, понимающие новые технологии хотя бы на уровне языков программирования. | Ограничение на выбор технологий. В IT-отделе есть Java-экспертиза (команда АБС + система кол-центра на Java Spring Boot). Python-экспертиза (скоринг). .NET-экспертиза (интернет-банк). React.js, PHP (сайт, кол-центр). |
| S3 | **Микросервисная архитектура для новых модулей интернет-банка** (в рамках задачи открытия депозитов). Модульность для дальнейшего расширения. | Архитектурно значимо: новый функционал ИБ реализуется как отдельные сервисы, а не расширение монолита. Это позволяет масштабировать независимо и использовать Kafka. |
| S4 | **Документирование системы для дальнейшего расширения.** Разработка документации для новых сервисов и интеграций. | Явное требование. Необходимо для команды цифровой трансформации и будущих этапов (полная автоматизация депозитов, расширение кредитного процесса). |
| S5 | **Реализация бизнес-функционала (СМС-подтверждение, депозиты) силами команды банка, без доработки ядра интернет-банка подрядчиком.** | Архитектурно значимо: новый функционал не должен зависеть от релизного цикла подрядчика. Подтверждает необходимость микросервисного подхода (S3) – вынести бизнес-логику за пределы монолита. |
| S6 | **Возможность расширения функционала депозитных ставок.** Модуль управления ставками должен поддерживать доступ для нескольких подразделений (бэк-офис депозитов и кредитов) с разграничением прав. | Требование безопасности из процесса: сотрудники депозитов и кредитов не должны иметь доступа к данным друг друга. Модуль ставок должен учитывать это ограничение. |
| **+R** | **+ Ограничения (Restrictions)** | |
| +R1 | **Шифрование трафика (TLS/HTTPS)** для всех каналов передачи данных: сайт, интернет-банк. | Обязательное требование безопасности. С сайта и из ИБ передаются персональные и финансовые данные. Стандарт для банковской отрасли. |
| +R2 | **Текущая платформа интернет-банка (ASP.NET MVC 4.5 / .NET Framework 4.5) несовместима с Kafka.** | Жёсткое ограничение. Новый функционал, требующий Kafka, должен быть реализован вне монолита ИБ – как отдельные микросервисы на совместимом стеке. |
| +R3 | **Kafka как брокер сообщений.** Если нужны очереди сообщений, бизнес предпочитает Kafka на перспективу. | Архитектурно значимо: определяет выбор брокера для асинхронных интеграций (заявки, уведомления, синхронизация данных). |
| +R4 | **Невозможно ускорить пакетный обмен между БД АБС и Кредитным конвейером.** Текущая интеграция АБС → КК через БД – 1 раз/сутки, изменить частоту нельзя. | Жёсткое ограничение. Для оперативных заявок (предодобренные кредиты, онлайн-депозиты) нужен обходной маршрут, минуя пакетный обмен через БД АБС. |
| +R5 | **Идентификация новых клиентов только в отделении.** Банк не может принимать новых клиентов без личной проверки документов. | Регуляторное ограничение. Заявки с сайта от новых клиентов всегда завершаются визитом в отделение. |
| +R6 | **Разграничение доступа между подразделениями депозитов и кредитов.** Сотрудники депозитного и кредитного подразделений не должны иметь доступа к данным друг друга (требование безопасности). | Влияет на проектирование модуля ставок и ролевой модели доступа в АБС. |
| +R7 | **Существующая интеграция ИБ с АБС – прямой доступ к БД.** Текущая архитектура интернет-банка предполагает прямое обращение к БД Oracle АБС. Это сохраняется для существующего функционала (платежи, текущие счета). | Для нового функционала (депозиты, кредиты) такой подход неприменим (R3). Но существующий функционал продолжает работать через прямой доступ к БД – миграция на новую схему интеграции выходит за рамки MVP. |
| +R8 | **Система кол-центра (платформа подрядчика) поддерживает расширение.** Архитектура микросервисная, в команде АБС есть Java-специалисты, способные дорабатывать систему. | Позитивное ограничение: систему кол-центра можно расширить для приёма заявок с сайта без полной зависимости от подрядчика. |
| +R9 | **Соблюдение требований регулятора по защите персональных данных.** Паспортные данные, ФИО, телефоны – персональные данные, подпадающие под ФЗ-152. | Дополнительное требование. Хранение и передача ПДн должны соответствовать законодательству. Влияет на выбор места хранения заявок и архитектуру сервисов. |
